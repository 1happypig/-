# Agent 信息整理工作流

> 当你需要整理某个**理论知识领域**的信息时，小德按照这个流程执行。
> 核心原则：**先计划确认，再轻量搜索；够用就好，不追求学术级严谨。**

---

## 三步流程

```
你提出主题
    │
    ▼
┌─────────────────────────────────┐
│  Step 1   计划确认               │
│           拆解子问题 → 确认边界   │
│           ⏸️  等你确认后再继续    │
└─────────────────────────────────┘
    │ (你确认后)
    ▼
┌─────────────────────────────────┐
│  Step 2   搜索 + 整理            │
│           轻量采集 → 去重筛选     │
│           → 按模板写成笔记        │
└─────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────┐
│  Step 3   自查纠错               │
│           对照检查清单逐项复核     │
│           → 修正问题后再输出      │
└─────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────┐
│  Step 4   输出归档               │
│           展示笔记 → 你确认无误    │
│           → 写入 Obsidian        │
└─────────────────────────────────┘
```

---

## Step 1：计划模式（Plan First）

### 目标
把"我想了解 X"变成有边界的研究计划，避免跑偏，也避免过度搜索浪费 token。

### 小德会做的事
1. **进入计划模式**，不直接搜索
2. 给出 2-4 个维度的拆解方案，例如：
   - 这个主题的核心概念是什么？
   - 有哪些主流方法论/流派？
   - 有什么公认的案例或实践？
3. **明确边界**：哪些方面要覆盖、哪些不需要（比如只看实践不看理论、只看近两年）
4. **等你确认**后才进入下一步

### 示例
```
你：「帮我整理暗黑like的终局玩法设计」

小德进入计划模式：
  研究方向：
  1. 终局玩法的常见类型（秘境/大秘境、PvP、世界Boss等）
  2. 各类型的核心设计目标和取舍
  3. 暗黑3/4、POE、Last Epoch 等代表性游戏的做法

  不覆盖：数值公式、服务器架构、商业化设计

  预计输出：一篇 2-3 节的结构化笔记，每节带简要案例

  确认这样？还是想调整？
```

---

## Step 2：搜索 + 整理（Search & Organize）

### 目标
在确认好的边界内收集信息，直接输出结构化笔记。

### 搜索策略
- **不并行多角度搜索**，按拆解的 2-4 个维度依次或合并搜索
- 优先找**高信息密度**的来源：行业博客、设计总结、分析文章
- 只有主题需要严谨性时才查论文/官方文档
- 搜到 3-5 篇优质文章即可，不追求穷举
- **每篇文章的链接都要保留**，整理时附到笔记末尾

### 信息筛选
- 同一观点有 2 个来源提及 → 保留
- 只有 1 个来源 → 保留但标注"单一来源"
- 明显 AI 生成/营销软文 → 丢弃
- 过时信息（游戏设计领域超过 3-5 年）→ 标注"可能需要更新"

### 输出模板

```markdown
---
tags: [领域, 子标签]
created: YYYY-MM-DD
---

# 标题

## TL;DR
- 3-5 条核心结论，看完就知道这笔记讲什么

## 1. 是什么
- 概念定义、边界

## 2. 怎么做 / 怎么用
- 方法论、流程、关键点

## 3. 案例
- 2-3 个实际例子

## 4. 我的理解（可选）
- 结合你的工作场景的思考

## 参考来源
- [文章标题](https://...) — 一句话说明这篇文章提供了什么信息
- [文章标题](https://...) — 一句话说明
```

> **硬性要求**：搜索时用到的每篇文章，都必须把链接附在「参考来源」下，方便你回溯原文。

---

## Step 3：自查纠错（Self-review）

### 目标
在给你看之前，小德自己先逐项检查一遍，把明显的错误拦下来。

### 检查清单
笔记写完后，逐项确认：

- [ ] **范围对齐**：笔记内容是否在 Step 1 确认的边界内？有没有跑题或遗漏？
- [ ] **事实校验**：数据和具体说法是否与搜索到的原文一致？有没有添油加醋？
- [ ] **观点标注**：主观判断是否标明了是"观点/社区评价"而非事实？
- [ ] **来源完整**：每篇引用的文章链接都附在「参考来源」下了吗？
- [ ] **路径正确**：文件要写到哪个文件夹？（理论知识 → `理论知识/`，游戏拆解 → 对应文件夹）
- [ ] **内部一致**：不同段落之间有没有自相矛盾的说法？

### 发现问题的处理
- 小问题（错别字、链接漏了）→ 直接修正，不提
- 中等问题（某条信息只有一个来源且不确定）→ 修正后在预览时标注 ⚠️
- 大问题（核心结论与来源不符）→ 修正后在预览时主动说明

---

## Step 4：输出 + 归档

### 小德会做的事
1. 把笔记给你**预览**，确认结构没问题
2. 写入 Obsidian **对应文件夹**（理论知识 → `理论知识/`，地图拆解 → 对应游戏文件夹，不要扔根目录）
3. 自动链接 vault 中的已有相关笔记
4. 如果有 MOC 索引，补上链接

### 不会做的事
- 不会写 20 页的报告
- 不会标注 15 个脚注
- 不会逐条做三方交叉验证

---

## 使用方式

| 你说的 | 小德的行为 |
|--------|-----------|
| 「整理一下 X」 | 进入计划模式 → 确认 → 搜索 → 输出 |
| 「快速查一下 X」 | 跳过计划模式，直接搜一段简答 |
| 「深入 X」 | 计划模式下多搜几轮，更详细 |

## 设计思路

这个简化版保留了 Deep Research 模式最有价值的两个环节：
- **先计划再动手**（来自 NotebookLM、NVIDIA AI-Q 的 evidence-grounded planning）
- **输出前预览**（减少返工）

去掉的：
- 5 角度并行搜索（token 大户，你的学习笔记不需要这么广的覆盖）
- 三方交叉验证（你不需要学术级严谨，2 源印证就够了）
- 渐进式摘要五层处理（过度加工，直接写干净的结构化笔记更高效）
